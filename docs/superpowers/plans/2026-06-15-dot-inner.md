# `dot_inner` Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add `A("…;…").dot_inner(B("…;…"))` -- a lazy binary expression that computes the per-outer-cell inner dot product of two tensor-of-tensor (ToT) operands over a general outer product, producing a plain tensor-of-scalars (T) result -- so production code stops routing ToT*ToT→T through `TA::einsum`.

**Architecture:** Reuse the existing general-product contraction engine verbatim for all outer bookkeeping (contraction/Hadamard/external) and the existing inner flat-dot kernel (`cont_engine.h:1542-1580`). The only genuinely new pieces are (1) result-type deduction that yields a plain `Tensor<S>` from ToT operands, and (2) a `ContractReduce`/element-op mode where the *result tile element is a scalar* while operands are nested -- so the inner dot writes `result += acc` directly instead of into a `[1]` cell that `einsum` later squeezes. `einsum`'s `DeNest::True` path is retained as the correctness oracle.

**Tech Stack:** C++20, TiledArray expression DSL (`src/TiledArray/expressions/`), MADWorld runtime, Boost.Test (`tests/ta_test`). Build via CMake/CTest.

---

## Strategy: native scalar result, vertical-slice first

We produce a **plain `Tensor<S>`** result tile directly -- no `[1]` inner cells, no squeeze. The heavy machinery is **verified to already support this**:

- `Tensor::gemm(left, right, gemm_helper, elem_muladd_op)` (`tensor.h:3220-3227`) is SFINAE'd only on `is_invocable_r_v<…, value_type&, const U&, const V&>` -- **no same-element-type assertion**. With `this` a `Tensor<S>`, the result cell `*(this_data+c)` is a scalar `S&` and `left/right` cells are inner tensors, so our scalar-dot op `void(S&, const InnerTensor&, const InnerTensor&)` fits. On an empty result it allocates a flat `Tensor(make_result_range(...))` (`tensor.h:3249-3252`) -- a contiguous scalar buffer.
- `ContractReduce::operator()` already routes the non-plain case to exactly this `gemm(result, left, right, gemm_helper(), elem_muladd_op())` (`contract_reduce.h:431`).
- The outer SUMMA, dist-eval, and `SparseShape` norm-GEMM operate on the *outer* structure and tile norms, independent of the result element type.

The **only hard blocker** is the taxonomy assert in `ContractReduceBase` (`contract_reduce.h:71-84`): today `mixed_tensors` (not all-plain, not all-nested) is asserted to imply a *nested* result. Our case -- both operands nested, result a scalar element -- trips it. So the contained new work is exactly three things:

1. **`ContractReduceBase` taxonomy** -- admit "both nested → plain scalar result" (the denest case) and relax the static_assert (Task 1, Step 6). *This is required, not conditional.*
2. **`ContEngine` inner-op branch** -- install the scalar-dot `elem_muladd_op_` when the result element is scalar (Task 1, Step 5), mirroring the phantom-unit branch at `cont_engine.h:1542-1580` but writing `result += acc` to a scalar.
3. **Node/engine/trait wiring** -- `DotInnerExpr`/`DotInnerEngine`/`ExprTrait` so the result type deduces to `Tensor<S>` and the engine forces the scalar branch (Steps 3-4, 7).

C++ expression-template nodes are all-or-nothing to compile (node + trait + engine + element op must coexist before anything type-checks), so **Task 1 lands the full vertical slice** -- all three pieces plus one `einsum` cross-check -- and Tasks 2+ add coverage incrementally with per-feature TDD against the now-existing scaffold.

Engine-internal steps state a **target implementation** plus the **acceptance test that gates it**; iterate against the compiler until green.

- [ ] **Step 3: Create the node header** (`dot_inner_expr.h`)

```cpp
#ifndef TILEDARRAY_EXPRESSIONS_DOT_INNER_EXPR_H__INCLUDED
#define TILEDARRAY_EXPRESSIONS_DOT_INNER_EXPR_H__INCLUDED

#include "TiledArray/expressions/binary_expr.h"
#include "TiledArray/expressions/dot_inner_engine.h"
#include "TiledArray/tensor/type_traits.h"
#include "TiledArray/type_traits.h"

namespace TiledArray::expressions {

namespace detail {
/// Result tile of dot_inner: a PLAIN Tensor<S>, S = inner numeric type of the
/// ToT operands (NOT result_of_mult_t, which would preserve nesting).
template <typename LEval, typename REval>
struct dot_inner_result {
  static_assert(TiledArray::detail::is_tensor_of_tensor_v<LEval> &&
                    TiledArray::detail::is_tensor_of_tensor_v<REval>,
                "dot_inner requires tensor-of-tensor operands");
  using inner_tile = typename LEval::value_type;           // e.g. Tensor<S>
  using S = TiledArray::detail::numeric_t<inner_tile>;
  using type = TA::Tensor<S>;
};
template <typename LEval, typename REval>
using dot_inner_result_t = typename dot_inner_result<LEval, REval>::type;
}  // namespace detail

template <typename Left, typename Right>
class DotInnerExpr;

template <typename Left, typename Right>
struct ExprTrait<DotInnerExpr<Left, Right>> {
  typedef Left left_type;
  typedef Right right_type;
  typedef detail::dot_inner_result_t<
      typename EngineTrait<typename ExprTrait<Left>::engine_type>::eval_type,
      typename EngineTrait<typename ExprTrait<Right>::engine_type>::eval_type>
      result_type;
  typedef DotInnerEngine<typename ExprTrait<Left>::engine_type,
                         typename ExprTrait<Right>::engine_type, result_type>
      engine_type;
  typedef TiledArray::detail::numeric_t<result_type> numeric_type;
  typedef TiledArray::detail::scalar_t<result_type> scalar_type;
};

template <typename Left, typename Right>
class DotInnerExpr : public BinaryExpr<DotInnerExpr<Left, Right>> {
 public:
  typedef DotInnerExpr<Left, Right> DotInnerExpr_;
  typedef BinaryExpr<DotInnerExpr_> BinaryExpr_;
  typedef typename ExprTrait<DotInnerExpr_>::left_type left_type;
  typedef typename ExprTrait<DotInnerExpr_>::right_type right_type;
  typedef typename ExprTrait<DotInnerExpr_>::engine_type engine_type;

  DotInnerExpr(const DotInnerExpr_&) = default;
  DotInnerExpr(DotInnerExpr_&&) = default;
  ~DotInnerExpr() = default;
  DotInnerExpr_& operator=(const DotInnerExpr_&) = delete;
  DotInnerExpr_& operator=(DotInnerExpr_&&) = delete;

  DotInnerExpr(const left_type& left, const right_type& right)
      : BinaryExpr_(left, right) {}
};

}  // namespace TiledArray::expressions

#endif  // TILEDARRAY_EXPRESSIONS_DOT_INNER_EXPR_H__INCLUDED
```

- [ ] **Step 5: Add the scalar-result inner-op branch to `ContEngine`** (`cont_engine.h`, alongside the phantom-unit branch at `:1542-1580`) -- target implementation

When the result tile element is a scalar (key off the `denest_to_scalar` tag and/or `!is_tensor_v<result_tile_element_type>`), install the scalar-dot element op:

```cpp
// dot_inner denest: operands are ToT, result element is a scalar S. The inner
// modes are fully contracted into a flat non-conjugating dot accumulated
// directly into the scalar result element (no [1] cell, no squeeze). Mirrors
// the phantom-unit branch but writes a scalar instead of a [1] cell.
const scalar_type factor = this->factor_;
this->element_nonreturn_op_ =
    [factor](result_tile_element_type& result,        // S& (scalar)
             const left_tile_element_type& left,       // inner Tensor
             const right_tile_element_type& right) {   // inner Tensor
      if (left.empty() || right.empty()) return;
      const std::size_t n = left.range().volume();
      TA_ASSERT(n == right.range().volume());
      const auto* lp = left.data();
      const auto* rp = right.data();
      result_tile_element_type acc{0};
      for (std::size_t j = 0; j < n; ++j) acc += lp[j] * rp[j];
      result += static_cast<result_tile_element_type>(factor) * acc;
    };
```

The dot loop is identical to `cont_engine.h:1571`; only the destination changes (`result += …` vs `result.data()[0] += …`). The operand inner permutation (e.g. `a,b,c` vs `b,c,a`) must already be normalized before the flat read -- see Task 3, which validates this.

- [ ] **Step 6: Extend `ContractReduceBase` taxonomy for nested-args → plain result** (`contract_reduce.h:71-84`) -- required

Today the taxonomy forbids our case:

```cpp
static constexpr bool mixed_tensors = !plain_tensors && !nested_tensors;
static_assert(!mixed_tensors || (mixed_tensors &&
                  is_nested_tensor_v<result_value_type>),
              "...must produce a nested tensor...");   // FIRES for us
```

For dot_inner `left_value_type`/`right_value_type` are nested (inner tensors) but `result_value_type` is scalar `S`, so `plain=false`, `nested=false`, `mixed=true`, `is_nested_tensor_v<S>=false` → assert fires. Introduce a distinct category and relax the assert, e.g.:

```cpp
// both operands nested, result a scalar element: a nesting-reducing (denest)
// contraction -- the inner modes are fully reduced to a scalar per outer cell.
static constexpr bool denest_to_scalar =
    TiledArray::detail::is_nested_tensor_v<left_value_type, right_value_type> &&
    !TiledArray::detail::is_nested_tensor_v<result_value_type>;
static_assert(!mixed_tensors || denest_to_scalar ||
                  is_nested_tensor_v<result_value_type>,
              "ContractReduce mixed plain/nested operands must produce a "
              "nested tensor, unless both operands are nested and the result "
              "is a scalar element (dot_inner denest).");
```

`ContractReduce::operator()` already routes `!plain_tensors` to `gemm(result, left, right, gemm_helper(), elem_muladd_op())` (`contract_reduce.h:431`), and that `Tensor::gemm` (`tensor.h:3220-3227`) accepts our scalar-result/ToT-arg combination unchanged -- so **no `operator()` body change is expected here**. Verify the `arena_plan_storage_t<Tensor<S>, ToT, ToT>` and `is_contraction_arena_tot_v<Tensor<S>, ToT, ToT>` paths are inert for a plain result (they should be -- arena handling is for nested results); if either asserts, gate it on `!denest_to_scalar`.
